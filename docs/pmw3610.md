# PMW3610 sensor knowledge

Facts confirmed from the full PixArt datasheet
(PMS0003-PMW3610DM-SUDU-DS-R2.4-170914, 38 pages,
https://trackballs.eu/media/Nakabayashi/Digio2/PMW3610DM-SUDU.pdf — the commonly
mirrored 8-page version lacks everything below) and validated on real hardware
(XIAO nRF52840 + PMW3610, 3-wire SPI).

## Pixel array / frame capture

- Pixel array is **22x22 = 484 pixels** ("Pixel Address Map for 22x22").
- Two documented image download methods:
  1. **Frame Capture burst** (uses BURST_READ 0x12): write 0xF1→0x11,
     0xBA→0x41 (+300us), 0x10→0x32, 0x80→0x36, wait 10ms, then burst read from
     0x12. Not suitable for 3-wire shared-MOSI/MISO wiring with
     `disable-burst-read`.
  2. **PIXEL_GRAB register-by-register** (works on 3-wire wiring):
     ```
     1) 0x41 = 0xBA          # SPI clock on request
     2) 0x7F = 0xFF          # page 1
     3) 0xB4 = 0xD7          # page-1 magic enable
     4) 0x7F = 0x00          # page 0
     5) 0x41 = 0xB5          # SPI clock off
     6) 0x32 = 0x90          # test clock on
     7) 0x35 = 0x01          # arm pixel grab
     8) read 0x47 until bit1 set   (10ms waits)
     9) read 0x2D until bit2 set   (10ms waits)
     10) read 0x35 → byte: bit7 = PG_Valid, bits[6:0] = pixel
     11) 0x2D = 0x01
        repeat 9-11 x484
     12) read 0x47: bit0 set = complete frame read
     ```
- After either procedure, **a full reset (power-up reset + reconfigure) is
  required to resume navigation**.
- The naive "PixArt pixel-grab convention" (write 0x35 then poll 0x35 for
  PG_VALID) does **not** work on PMW3610 — PG_VALID never asserts without the
  arming sequence above (confirmed on hardware).
- **Hardware-validated additions beyond the datasheet text** (XIAO nRF52840 +
  PMW3610, 2026-07-04): a stationary/resting sensor updates its ready bits
  only once per sensor frame period, which can be minutes in REST3. Before
  arming pixel grab: (1) force RUN mode by setting PERFORMANCE (0x11)
  bits[7:4]=0xF, and (2) confirm the mode switch actually landed by clearing
  and polling OBSERVATION1 (0x2D) for the frame/mode bits — the force-run
  write only takes effect at the next frame boundary. Skipping either step
  gives 0-1 valid pixels per attempt depending on how deep into rest mode the
  sensor was. With both, all 484/484 pixels come back valid with `0x47` bit0
  (complete) set, in ~2s (~4ms/pixel, i.e. the run-mode frame period).
  22x22 confirmed both by datasheet Figure 17 and by measuring
  adjacent-row pixel correlation across candidate array sizes (22x22 scored
  0.76 vs 0.26–0.56 for 19–21).
- 0x32 (DTEST2_PAD) is shared with the runtime smart-algorithm tweak
  (0x00/0x80) — post-capture reset clears it.

## Other registers / behaviors (from driver work)

- Register writes generally need the SPI clock request wrapper: 0x41=0xBA,
  wait 300us, write, 0x41=0xB5 (see `pmw3610_write` in the driver).
- CPI is set via paged register RES_STEP (0x85) with the 0x7F page-select
  sequence; value = cpi/200 (200–3200).
- Motion burst read is at 0x12 (BURST_READ), 10 bytes:
  Motion, DX_L, DY_L, DXY_H, SQUAL, Shutter_Hi, Shutter_Lo, PixMax, PixAvg, PixMin.
- Product ID (0x00) = 0x3E; self-test = write 0x2D=0x00, wait, check bits[3:0]
  all set.
