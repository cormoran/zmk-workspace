# Renode ZMK bring-up log

Real entries from the bring-up session, newest at bottom. See
`references/renode-notes.md` for the distilled, reusable version of each
gotcha; this log is the chronological narrative.

## Setup

- `bash scripts/install_renode.sh` → downloaded Renode 1.16.1 portable to
  `~/.renode/1.16.1/renode`. Worked first try, no system mono/dotnet
  needed.
- Renode's bundled `platforms/cpus/nrf52840.repl` is a plain, board-agnostic
  nRF52840 (uart0 @0x40002000, uart1 @0x40028000, radio, flash 1MB, ram
  256KB) — matches the XIAO's actual silicon. Copied into
  `platforms/xiao_nrf52840.repl` for reproducibility.

## T0 boot

- First `west build` attempt from `zmk-feature-studio-rpc-perf/` without
  `-s dependencies/zmk/app` failed (`Unknown CMake command
  "zephyr_include_directories"`) — that repo's own root `CMakeLists.txt` is
  a Zephyr *module* manifest, not an application. Fixed by pointing `-s` at
  `dependencies/zmk/app` and adding `-DZMK_CONFIG=.../tests/zmk-config/config`
  + `-DZMK_EXTRA_MODULES="tests/zmk-config;<repo root>"` (shield lives under
  `tests/zmk-config`, custom RPC-perf code lives at the repo root).
- Second failure: `Could not find a package configuration file ... Zephyr-sdk`
  with a stack trace mentioning a *different* project's zephyr checkout
  (`zmk-feature-os-detection`) — a stale CMake user-package-registry entry.
  Fixed with explicit `ZEPHYR_TOOLCHAIN_VARIANT=zephyr` and
  `ZEPHYR_SDK_INSTALL_DIR=~/agent-home/zephyr-sdk-0.16.8`.
- Build succeeded (`studio-rpc-uart.overlay` v1: uart0=console, uart1=RPC).
  Booted in Renode via `platforms/single.resc` + monitor `-P` port. First
  `single.resc` had a multi-line `:description: >` block — broke the
  monitor tokenizer, resc silently never got included, no sockets ever
  opened. Fixed to a single-line description + `#` comments.
- Booted again: monitor came up, sockets opened, `start` issued — but
  **zero bytes ever appeared on either UART**, forever, even though
  `sysbus.cpu ExecutedInstructions` was climbing (CPU genuinely running).
  Root cause #1: `&qspi`/`p25q16h` (external NOR flash, enabled by default
  on the XIAO board dts) — Renode's QSPI peripheral is an SVD-register stub
  with no working `EVENTS_READY`, so `nordic_qspi_nor`'s driver init
  busy-waits forever before `main()`. Disabling `&qspi` alone wasn't
  enough — devicetree `status=disabled` doesn't cascade to the child
  `p25q16h` node, so `CONFIG_NORDIC_QSPI_NOR` still got built and still
  hung. Disabled both explicitly in the overlay — fixed.
- Rebuilt, rebooted: **still silent**. Root cause #2: the board's
  `board_cdc_acm_uart`/`&usbd` (USB CDC-ACM, enabled by default regardless
  of any snippet) — same class of bug: `usb_enable()`'s low-level
  `usbd_enable()` busy-waits on `USBD->EVENTCAUSE.READY`, which real
  silicon sets automatically a few µs after `ENABLE` (nothing to do with a
  host being attached) but Renode's USBD stub never sets. Confirmed with a
  targeted experiment: a build with `CONFIG_ZMK_USB=y` and *no* CDC-ACM
  node at all (pure ZMK HID class) hit the identical hang — so this is not
  CDC-ACM-specific, USB is unconditionally broken on this Renode nRF52840
  model. Also discovered the board's `BOARD_SERIAL_BACKEND_CDC_ACM` Kconfig
  cascade (`Kconfig.cdc_acm_serial.defconfig`, sourced unconditionally by
  `boards/seeed/xiao_ble/Kconfig.defconfig`) force-enables
  `CONFIG_USB_DEVICE_STACK`/`CONFIG_USB_DEVICE_INITIALIZE_AT_BOOT`
  regardless of `CONFIG_ZMK_USB`. Fixed with
  `-DCONFIG_ZMK_USB=n -DCONFIG_BOARD_SERIAL_BACKEND_CDC_ACM=n` plus
  disabling `&board_cdc_acm_uart`/`&usbd` in the overlay.
- Rebuilt, rebooted: console showed `# CONFIG_CONSOLE is not set`/no boot
  banner at all — without USB, nothing else on this board turns on
  `CONFIG_CONSOLE`/`CONFIG_LOG`/`CONFIG_UART_CONSOLE` by default. Added
  `-DCONFIG_LOG=y -DCONFIG_CONSOLE=y -DCONFIG_UART_CONSOLE=y
  -DCONFIG_UART_INTERRUPT_DRIVEN=y` explicitly.
- **T0 GREEN**: real console output —
  `*** Booting Zephyr OS build 9df4b12b5af3 ***` /
  `[00:00:00.000,244] <inf> zmk: Welcome to ZMK!` — captured over the
  uart0 socket.

## T1 Studio RPC over UART

- Compiled `zmk-studio-messages` protos with system `protoc` (3.12.4) +
  `--python_out`; needed `PROTOCOL_BUFFERS_PYTHON_IMPLEMENTATION=python`
  env var since the installed `protobuf` runtime (6.33.6) is too new for
  code generated by that old a `protoc` otherwise
  (`TypeError: Descriptors cannot be created directly`).
- Sent a `core.get_device_info` request over the RPC UART socket: no
  response, ever, no matter the timeout. Direct `sysbus.uart1 WriteChar`
  injection of the same framed bytes (bypassing the socket entirely): also
  nothing. Ruled out socket-layer issues this way.
- Traced it to `zmk_rpc`'s `refresh_selected_transport()`: the built-in
  UART transport is tagged `ZMK_TRANSPORT_USB` and only gets `rx_start()`
  called once `zmk_endpoint_get_selected().transport == ZMK_TRANSPORT_USB`,
  which requires `zmk_usb_is_hid_ready()` — a real, host-negotiated
  `USB_DC_CONFIGURED` state. Since USB is unconditionally broken under this
  Renode model (see T0), that can never happen — the console log
  confirmed: `get_selected_transport: No endpoint transport selected` /
  `Failed to select a transport!`. The UART, ISR, and framing code are all
  fine; RX is just never armed.
- Fix: added `renode-test-module/` — a small additive Zephyr module (no
  vendored ZMK files touched) registering a second UART transport
  (`src/renode_uart_transport.c`, near-identical to ZMK's own
  `uart_rpc_transport.c`) tagged `ZMK_TRANSPORT_NONE` instead of
  `ZMK_TRANSPORT_USB`. `ZMK_TRANSPORT_NONE` is exactly what
  `zmk_endpoint_get_selected()` naturally reports with USB and BLE both
  off and no persisted settings — confirmed by the same log line above.
  Gated behind `CONFIG_ZMK_RENODE_STUDIO_UART_TRANSPORT` (default n);
  `CONFIG_ZMK_STUDIO_TRANSPORT_UART=n` disables the real (here-unusable)
  transport so the two don't fight over the same UART's interrupt
  callback.
- Rebuilt, rebooted, sent `GetDeviceInfo` again: **real response**,
  correctly decoded: `request_response { request_id: 1 core {
  get_device_info { name: "MAK" } } }` — `"MAK"` matches
  `CONFIG_ZMK_KEYBOARD_NAME` for the `my_awesome_keyboard` shield exactly.
  **T1 first-request GREEN.**
- Sent a *second* request (`get_lock_state`) over the same still-open
  connection: timed out. Same for a third attempt on a brand new
  connection, and for a raw `sysbus.uart1 WriteChar`-injected request —
  ruling out "stale/reused socket" as the cause. `sysbus.cpu
  ExecutedInstructions` was climbing steadily throughout the "hang" (millions
  of instructions/2s) — not a halt, something spinning. Traced to the TX
  interrupt handler never calling `uart_irq_tx_disable()` once the TX ring
  buffer drains (this line is commented out in both ZMK's own
  `uart_rpc_transport.c` and our copy) — under Renode,
  `uart_irq_tx_ready()` reads as a level condition, so once the first
  response finishes sending, the CPU gets re-interrupted forever with
  nothing to send, permanently starving the `studio_rpc_thread` that would
  process the next request. Fixed (in our own transport file only) by
  explicitly disabling the TX IRQ once the buffer drains.
- Rebuilt, rebooted, sent three requests (`get_device_info`,
  `get_lock_state`, `get_device_info` again) over one connection: **all
  three succeeded**, correctly decoded, correct `request_id` echoed back
  each time. **T1 GREEN, robustly** — this is the state captured in
  `scripts/renode_test.py::test_t1_studio_rpc_uart`.

## T2 wired split

- Built central (`CONFIG_ZMK_SPLIT_ROLE_CENTRAL=y`) and peripheral
  artifacts with `split-wired-uart.overlay` (uart0=console, uart1=
  `zmk,wired-split` device) + `CONFIG_ZMK_SPLIT_WIRED_UART_MODE_INTERRUPT=y`
  (explicit, since the Kconfig `choice` has no `default` and would
  otherwise pick ASYNC/DMA mode first if `SERIAL_SUPPORT_ASYNC` is
  visible — untested under Renode, interrupt mode is the same well-tested
  path used for Studio RPC). Both builds succeeded first try.
- `platforms/split_wired.resc`: two `mach create`, `emulation
  CreateUARTHub "split_link"`, both machines' `uart1` connected to it,
  both `uart0` on their own console sockets.
- First boot: peripheral's own console showed it eagerly transmitting split
  events at t≈5ms (`split_peripheral_wired_report_event: Payload ... 5a 6d
  4b 77 ...` — `"ZmKw"` magic bytes) from its synthetic boot-time kscan
  presses. Central's console showed nothing about receiving them, ever
  (checked repeatedly over ~10s of real time, connection kept open the
  whole time). No crash, no error.
- Diagnosed with `sysbus.gpio0 OnGPIO 2 true`/`false` on the peripheral,
  injected ~3 real seconds after `start` (well after both sides' `SYS_INIT`
  should have completed): central's console immediately showed a fresh
  `position: 0` key-press/release cycle at the corresponding virtual
  timestamp. **The hub link works correctly** — the earlier "central never
  receives anything" was a boot-time race (peripheral's two synthetic
  events fired before central's `uart_irq_rx_enable()` in
  `zmk_split_wired_central_init`'s `SYS_INIT` had necessarily run yet;
  there's no cross-machine execution-order guarantee at t=0), not a hub
  wiring bug.
- **T2 GREEN**: `scripts/renode_test.py::test_t2_split_wired` boots both
  machines, waits 3s for settle, injects the GPIO toggle, and asserts
  `"position: 0"` appears in the central's console within 10s.

## T3 BLE (experimental, capped)

- Single board, `CONFIG_ZMK_BLE=y`, no peer, no Renode BLE medium wiring at
  all (simplest possible case): booted, then ~10s later:
  `bt_settings: settings_subsys_init failed (err -33)` →
  `zmk: BLUETOOTH FAILED (-33)` → ~10s after that, `ASSERTION FAIL [err ==
  0] ... Controller unresponsive, command opcode 0x1009 timeout with err
  -11` → `ZEPHYR FATAL ERROR 3: Kernel oops on CPU 0` → system halts.
- Renode's own bundled multi-node Zephyr BLE examples
  (`nrf52840-ble-zephyr.resc`, using `CreateBLEMedium`) suggest the radio
  peripheral itself isn't fundamentally broken in Renode — this looks like
  a ZMK-specific settings/controller-bringup interaction, not "BLE doesn't
  work in Renode" categorically. Did not chase further (explicitly capped
  per the mission's T3 time-box).
- Captured as `scripts/renode_test.py::test_t3_ble_single_board_boot`,
  skipped by default (`RENODE_ZMK_RUN_T3=1` to run) since it documents a
  known failure rather than a passing capability. Verified it does
  reproduce the exact failure signature (`ZEPHYR FATAL ERROR` within 20s)
  on a real run.

## Harness bugs found while writing `renode_test.py` (not Renode/ZMK issues)

- `RenodeSession.start()` built the `i @<resc>` monitor command from just
  `resc_path.name` (e.g. `"single.resc"`) instead of the path relative to
  the process's cwd (`"platforms/single.resc"`) — silently failed to find
  the file (same failure mode as the earlier manual multi-line-description
  bug: monitor comes up, script inclusion silently doesn't happen, every
  `connect_uart()` call times out with `Connection refused` no matter how
  long you wait). Fixed to use `resc_path.relative_to(SKILL_DIR)`.
- Using fixed port numbers per test meant a Renode process leaked by an
  *earlier failed* run (because `addCleanup(session.stop)` was registered
  too late — after `connect_uart()`, which is exactly the call that was
  failing) kept squatting on that port. The next run's client then
  connected to the *stale* process's monitor (which happily accepted
  `start`) instead of its own fresh one — and the stale process's UART
  sockets never existed, so `connect_uart()` timed out identically,
  masking the real (already-fixed) bug and looking like it hadn't been
  fixed at all. Fixed by (a) registering `addCleanup(session.stop)`
  immediately after `session.start()` succeeds, before any
  `connect_uart()` call, and (b) randomizing the port base per test run.
- `wait_for_text()`/`drain_text()` were called with `console_sock.sock`,
  but `RpcSocket` (from `rpc_client.py`) exposes the underlying raw socket
  as `._sock`, not `.sock` — `AttributeError`. Fixed the call sites.
- The T3 probe test composed its cmake arg list as `[..., "-DCONFIG_ZMK_BLE=y",
  *build_fw.COMMON_ARGS]` — `COMMON_ARGS` itself contains
  `-DCONFIG_ZMK_BLE=n` (for the other tiers), and cmake `-D` flags are
  last-wins, so the explicit `y` got silently overridden back to `n` by
  the later `COMMON_ARGS` entry. The resulting "BLE" build genuinely
  never touched Bluetooth at all (confirmed via `.config`:
  `# CONFIG_ZMK_BLE is not set`) and of course booted cleanly — which
  looked like "BLE works now!" until checking `.config`. Fixed by putting
  the override *after* `COMMON_ARGS` in the arg list.
